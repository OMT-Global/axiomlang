# Stage1 LSP analyzer

`axiomc lsp` starts the bounded `axiom-analyzer` Language Server Protocol
endpoint over stdio.

This is an analyzer scaffold for issue #241, not the full editor feature set.
It intentionally ships only the protocol surface that can be backed by the
current compiler diagnostics:

- JSON-RPC 2.0 messages framed with `Content-Length`.
- `initialize`, `initialized`, `shutdown`, and `exit`.
- `textDocument/didOpen` and full-document `textDocument/didChange`.
- `textDocument/publishDiagnostics` generated by the stage1 parser, HIR type
  checker, capability checks, and MIR lowering path.

Unsupported request methods receive JSON-RPC `-32601`.

## Running

```bash
cargo run --manifest-path stage1/Cargo.toml -p axiomc -- lsp
```

Editor clients should configure `axiomc lsp` as a stdio language server for
`.ax` files.

## Current boundaries

The server advertises full-document synchronization and analyzes the current
in-memory document text sent by the editor. Ranged incremental
`didChange` payloads are ignored until document-state tracking and range patch
application are implemented. It
does not yet build the full package graph from `axiom.toml`, so diagnostics for
cross-file imports and workspace capability context remain owned by
`axiomc check`. It also does not yet provide hover, goto-definition,
completion, rename, semantic tokens, code actions, or multi-error recovery.

Those features remain part of the broader #241 scope and depend on the
diagnostic recovery work called out in that issue.
